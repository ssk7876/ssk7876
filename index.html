🔍 문제 원인 분석

핵심 원인: Supabase의 RLS(Row Level Security) 정책

앱에서 사용하는 키는 anon (익명) 키입니다. Supabase는 기본적으로 anon 키로는 DELETE가 차단됩니다. RLS 정책이 설정되어 있지 않거나, DELETE를 허용하는 정책이 없으면 삭제가 조용히 실패합니다.

테이블 에디터에서도 안 된다 → 이건 Supabase 대시보드 자체에서도 RLS가 활성화된 채로 DELETE 정책이 없다는 뜻입니다.

✅ 해결 방법 (Supabase 대시보드에서 직접 설정)

방법 1: RLS를 비활성화 (간단하지만 보안 주의)

Supabase 대시보드 → Table Editor → 각 테이블 → RLS 비활성화

또는

방법 2: DELETE 허용 RLS 정책 추가 ✅ (권장)

Supabase 대시보드 → Authentication > Policies 에서 아래 SQL을 실행하세요:

SQL Editor에 아래를 붙여넣고 실행:

sql
-- sf_users 테이블 DELETE 허용
CREATE POLICY "allow_delete" ON sf_users
FOR DELETE USING (true);

-- sf_disinfection

FOR DELETE USING (true);

-- sf_seedling

-- sf_sowing

FOR DELETE USING (true);

-- sf_harvest
CREATE POLICY "allow_delete" ON sf_harvest
FOR DELETE USING (true);

-- sf_disaster
CREATE POLICY "allow_delete" ON sf_disaster
FOR DELETE USING (true);

-- sf_extra_varieties

FOR DELETE USING (true);

방법 3: RLS 자체를 끄기 (가장 빠름)

sql
ALTER TABLE sf_users DISABLE ROW LEVEL SECURITY;
ALTER TABLE sf_disinfection DISABLE ROW LEVEL SECURITY;
ALTER TABLE sf_seedling DISABLE ROW LEVEL SECURITY;

RLS 비활성화란?

RLS = Row Level Security = 행 수준 보안

쉽게 비유하면 이렇습니다:

🏢 회사 건물로 비유

Supabase 데이터베이스를 회사 건물, 각 테이블을 사무실이라고 생각하면:

  • RLS 활성화 (켜짐) = 사무실마다 잠금장치가 있고, 출입증(권한)이 있는 사람만 들어갈 수 있음
  • RLS 비활성화 (꺼짐) = 잠금장치를 열어두는 것. 누구든 들어올 수 있음

🔑 지금 상황

앱은 anon 키 (익명 방문자 출입증)를 사용합니다.

그런데 Supabase는 기본적으로:

  • 읽기(SELECT) ✅ 허용
  • 쓰기(INSERT/UPDATE) ✅ 허용
  • 삭제(DELETE) ❌ 차단 ← 여기서 막힘!

그래서 회원 퇴출 버튼을 눌러도 앱 화면에서만 사라지고, 실제 DB에는 그대로 남아있는 상태입니다. 앱을 새로 열면 다시 불러와서 부활하는 것처럼 보이는 거고요.

🔓 비활성화하면?

잠금장치를 아예 풀어서 삭제도 자유롭게 되게 하는 것입니다.

보안 걱정이 되실 수 있는데, 이 앱은 Supabase 자체 로그인을 쓰는 게 아니라 앱 안에서 자체적으로 아이디/비밀번호를 관리하기 때문에, RLS를 꺼도 실제 사용에는 문제가 없습니다.
